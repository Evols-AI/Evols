# Communication Drafter

## Purpose

Drafts messages calibrated to audience, channel, and stakes. Pulls from work context for stakeholder details and relationship dynamics. Integrates with the Say No Playbook when messages involve pushback.

---

## Channel Calibration

| Channel | Tone | Length | Structure |
|---------|------|--------|-----------|
| **Slack DM** | Casual, direct | 1-3 sentences | No headers, no bullets |
| **Slack Channel** | Professional, concise | 3-5 sentences, or bullets | Optional bold headers for sections |
| **Email to Peer** | Collegial, clear | 1-2 short paragraphs | Clear ask at top |
| **Email to Manager** | Direct, data-oriented | 2-3 paragraphs | Context -> Ask -> Next steps |
| **Email to Executive** | Concise, impact-focused | 3-5 sentences max | Lead with the so-what |
| **Status Update** | Structured, scannable | Bullets | Green/Yellow/Red or equivalent |
| **Document** | Thorough, narrative | Full doc | Use Document Accelerator skill |

## Drafting Process

### Step 1: Clarify the Goal

Before drafting, confirm:
- **What's the ask?** (Inform, request action, get decision, give update, push back?)
- **Who's the audience?** (Check audience profiles if available)
- **What channel?** (Determines tone and length)
- **What's the stake level?** (Routine / Important / High-stakes)

### Step 2: Draft

**Structure by message type:**

**Informing / Updating:**
> [One-line summary of what happened]
> [2-3 sentences of context if needed]
> [What this means for the recipient / what's next]

**Requesting Action:**
> [What you need, specifically]
> [Why it matters / context]
> [Timeline]

**Seeking Decision:**
> [The decision that needs to be made]
> [Options with tradeoffs, briefly]
> [Your recommendation]
> [What you need from them and by when]

**Pushing Back:** (Integrate with Say No Playbook)
> [Acknowledge the request]
> [The constraint]
> [Your alternative / redirect / deferral]

### Step 3: Calibrate

- Check against audience profile -- adjust detail level, tone, and structure
- For high-stakes messages: read aloud test -- does it sound like something the user would actually send?
- For manager communications: lead with what they care about, not what you did
- For executive communications: ruthlessly cut -- every sentence must earn its place

## Quick Templates

### Status Update (Any Format)
```
**[Project Name] -- [Date]**
**Status:** [Green/Yellow/Red]
**Progress:** [What happened since last update]
**Next:** [What's coming]
**Risks/Blockers:** [If any -- otherwise omit this line]
**Ask:** [If any -- otherwise omit this line]
```

### Async Decision Request
```
**Decision Needed:** [What needs to be decided]
**By:** [Date/time]

**Context:** [2-3 sentences]

**Options:**
1. [Option A] -- [Tradeoff]
2. [Option B] -- [Tradeoff]

**My recommendation:** [Option] because [reason]

Please reply with your preference or concerns by [date].
```

### Meeting Follow-Up
```
Thanks all -- here's what we landed on:

**Decisions:**
- [Decision 1]
- [Decision 2]

**Action items:**
- [Name]: [Task] by [date]
- [Name]: [Task] by [date]

**Open questions:**
- [Question] -- [Who will resolve] by [when]

Let me know if I missed anything.
```

## Rules

1. **Never draft in a vacuum.** Always check work context for relationship dynamics and stakeholder preferences.
2. **Shorter is better.** The user's credibility increases with concision.
3. **Lead with the ask.** Especially for managers and executives -- don't bury the point.
4. **Match the user's voice.** Drafts should sound like the user, not like a template. Adjust based on their typical communication style.
5. **Flag high-stakes messages.** For anything going to a VP+ or involving conflict, ask the user to review before sending.
